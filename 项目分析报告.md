## 一、项目概述

### 1.1 核心功能与目标

本项目是一个基于虚构双月行星系统的**完整历法引擎与可视化日历应用**。其世界观设定为一颗质量为 2 倍地球的宜居行星（"地星"），拥有两颗以 2:1 轨道共振的卫星——主月"塞勒涅"（朔望月 19.5883 天）和副月"恩底弥翁"（朔望月 41.4948 天）。

项目的核心目标是：
- 实现一套**三轨协同历法体系**：阳历（太阳回归年）定四季 → 主月历（塞勒涅历）定纪日 → 副月历（恩底弥翁历）定潮汐
- 提供天文事件自动检测（二十四节气、月相、双月合朔/冲日）
- 提供潮汐等级每日计算（大潮/中潮/小潮）
- 以 Web 日历应用形式进行可视化呈现

### 2.2 核心算法分析

#### 2.2.1 阳历系统 (solar.ts)

**闰年判定**（8 年 5 闰 + 600 年修正）：

```
基本规则：year % 8 ∈ {1, 2, 4, 5, 7} → 闰年（350天）
          year % 8 ∈ {0, 3, 6}       → 平年（349天）
修正规则：year % 600 === 0          → 强制平年
```

代码实现（`src/engine/solar.ts:13-23`）：

```typescript
export function isLeapYear(year: number): boolean {
  const remainder = year % 8
  const isBasicLeap = [1, 2, 4, 5, 7].includes(remainder)
  // 600年修正
  if (isBasicLeap && year % 600 === 0) return false
  return isBasicLeap
}
```

**月份天数分配**：1 月 = 30 天，2-11 月各 29 天，12 月 = 29/30 天。

**solarToAbs 转换流程**：
1. 输入校验（年/月/日合法性）
2. 累加之前所有年份的总天数（利用缓存 `_yearStartCache`）
3. 累加当年内之前月份的天数
4. 加上当月日期（转为 0-based 索引）

**absToSolar 逆转换**：
1. 使用平均年长（349.6266）估算年份
2. 逐年校正（最多 5 次迭代）
3. 计算年内偏移，分配月份

#### 2.2.2 主月历系统 (lunar-primary.ts)

**月份天数**：通过**累积舍入法**（Cumulative Rounding）计算：

```typescript
// 第 N 个全局月的起始 ABS
function cumulativeDaysBeforeMonth(globalMonthIndex: number): number {
  return Math.round(globalMonthIndex * SYNODIC_MONTH)  // 19.5883
}

// 该月天数 = 下月起始 - 本月起始
function getLunarPrimaryMonthDays(globalMonthIndex: number): number {
  return cumulativeDaysBeforeMonth(globalMonthIndex + 1) -
         cumulativeDaysBeforeMonth(globalMonthIndex)
}
```

此方法保证长期平均月长精确等于 19.5883 天，且每月天数在 19-20 之间。

**无中气置闰法**：遍历每个主月包含的所有日期，检查是否有偶数位节气（中气）。无中气的月份标记为闰月。这是与中国农历一致的规则，完全贴合天文观测。

**月份序号推进**：含中气 → `monthNum++`；不含中气 → `monthNum` 不变（闰月复用上个月序号）。

**月龄计算**：

```typescript
export function calculateMoonAge(abs: number, cycleLength: number): number {
  return ((abs % cycleLength) + cycleLength) % cycleLength
}
```

假设 ABS=0 时主月恰处"朔"（月龄=0），此后每日的月龄由 ABS 对朔望月长度取模得到。

#### 2.2.3 副月历系统 (lunar-secondary.ts)

与主月历采用**相同的累积舍入法**计算月份天数，每月在 41-42 天之间。

**关键差异**：
- 使用副月朔望月周期 41.4948 天
- `resolveYearMonth` 使用简化逻辑（不检查中气），仅按 ABS 是否越过阳历年起点的月份计数
- **未实现置闰**：`isLeapMonth` 恒为 `false`

#### 2.2.4 潮汐计算 (astronomical.ts:calculateTideLevel)

这是项目中最为核心且复杂的算法，基于**双月会合周期**（37.1040 天）计算每日潮汐等级。

**物理原理**：
- 双月的引潮力叠加效应取决于两者的黄经差
- 合（0°黄经差）或冲（180°黄经差）→ 引潮力叠加最大 → **大潮**
- 弦向垂直（90°或 270°黄经差）→ 引潮力部分抵消 → **中潮**
- 其他位置 → 引潮力一般 → **小潮**

**算法流程**（`src/engine/astronomical.ts:160-182`）：

```
Step 1: 计算当日 ABS 在双月会合周期中的相位位置
        phaseInCycle = ((abs % 37.1040) + 37.1040) % 37.1040

Step 2: 折叠到半周期范围 [0, 18.552]
        normalized = min(phaseInCycle, 37.1040 - phaseInCycle)

Step 3: 计算距合/冲位置的最短距离
        distanceFromAlignment = min(normalized, |normalized - 18.552|)

Step 4: 分级判定
        if distanceFromAlignment ≤ 3        → 大潮 (SPRING)
        elif |normalized - 9.276| ≤ 4.5     → 中潮 (NEAP)
        else                                → 小潮 (NORMAL)
```

**判定阈值**：
- 大潮窗口：距合/冲位置 ±3 天（约合 ±29° 黄经差）
- 中潮窗口：距弦向垂直位置 ±4.5 天（约合 ±44° 黄经差）

### 2.3 三轨协同机制

阳历、主月历、副月历三者通过以下机制协同工作：

```
                     ┌──────────────────────┐
                     │   AbsoluteDayNumber  │
                     │     (统一坐标基准)     │
                     └──────┬───────────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                  ▼
    ┌──────────┐    ┌──────────────┐    ┌──────────────┐
    │ 阳历      │    │ 主月历        │    │ 副月历        │
    │ (Solar)   │    │ (Primary)     │    │ (Secondary)   │
    ├──────────┤    ├──────────────┤    ├──────────────┤
    │年: 0-5000 │    │月: 大20/小19  │    │月: 大42/小41  │
    │月: 1-12   │    │闰: 无中气置闰  │    │闰: (未实现)   │
    │日: 29/30  │    │月相: 8阶段     │    │月相: 8阶段    │
    │季: 4季    │    │               │    │              │
    │节气: 24个 │    │               │    │              │
    └─────┬────┘    └──────┬───────┘    └──────┬───────┘
          │                │                   │
          └────────────────┼───────────────────┘
                           ▼
              ┌─────────────────────┐
              │   统一聚合 API       │
              │  getCalendarDate()   │
              ├─────────────────────┤
              │  + 星期 (5日制)      │
              │  + 干支纪日 (60循环) │
              │  + 天象事件检测      │
              │  + 潮汐等级计算      │
              └─────────────────────┘
```

**协同关键点**：

1. **中气绑定**：主月历的闰月判定依赖阳历的中气（二十四节气的偶数位），这形成了阳历与主月历之间的刚性约束
2. **年份同步**：主月历的年份划分以阳历年起点的 ABS 为边界，确保 `lunarPrimary.year === solar.year`
3. **潮汐独立**：潮汐等级完全基于双月会合周期（37.1040 天），与阳历、主月历无直接耦合，但通过 ABS 坐标与三者统一
4. **预计算缓存**：`precomputeCache()` 对所有 ABS 执行 `getCalendarDate()`，一次性完成三轨数据的同步填充

## 三、潜在问题与优化建议

### 3.1 ★★★ 严重：阳历平年天数与回归年不匹配

**问题描述**（`src/engine/constants.ts:32`）：

```typescript
DAYS_IN_COMMON_YEAR: 349,  // ✅ 已修正（原错误值为350）
DAYS_IN_LEAP_YEAR: 350,
```

代码注释表明**有人将 `DAYS_IN_COMMON_YEAR` 从 350 改为 349**，理由是"修正"。然而，这一修改是错误的。

**数学验证**：

| 参数 | 值 |
|------|-----|
| 回归年长度（TROPICAL_YEAR） | 350.6266 天 |
| 当前 8 年 5 闰平均年长 | (3×349 + 5×350) / 8 = **349.625 天** |
| 误差 | 349.625 - 350.6266 = **-1.0016 天/年** |
| 8 年累积误差 | **约 -8 天** |

以 349/350 为平年/闰年天数，每年平均比回归年少约 **1 天**。这导致历法每年系统性漂移 1 天，完全无法满足农业四季定位需求。

**设计文档中的矛盾**（`历法.md`）：

- 文档正文说"平年天数：349 天，闰年天数：350 天"
- 但公式给出 "(3×350 + 5×351)/8 = 350.625"，暗示平年应为 350、闰年应为 351

**建议**：将 `DAYS_IN_COMMON_YEAR` 改回 **350**，`DAYS_IN_LEAP_YEAR` 改为 **351**。相应的月份分配需调整为：1 月 = 30 天，2-11 月各 29 天，12 月 = 30（平年）/ 31（闰年）。

### 3.2 ★★★ 严重：月相区间存在空隙

**问题描述**（`src/engine/constants.ts:84-113`）：

月相区间使用整数边界定义，例如主月：
```
朔: [0, 2]  → 盈月: [3, 7]  → 上弦: [8, 12]  ...
```

月龄通过 `abs % 19.5883` 计算，会产生**连续的分数值**。月龄 2.3、7.5、12.8 等落在整数间隙中，不匹配任何区间。此时 `mapMoonAgeToPhase` 回退到 `MoonPhaseEnum.NEW_MOON`（朔），导致错误标记。

**影响**：在所有空隙日，用户看到的月相图标会错误地显示为 🌑（朔）。

**建议**：将区间改为连续覆盖，例如：
```typescript
{ name: 'new-moon', startAge: 0, endAge: 2.5 },
{ name: 'waxing-crescent', startAge: 2.5, endAge: 7.5 },
// ... 依此类推
```

或者采用"计算到最近相位中心距离"的方式替代区间映射。

### 3.3 ★★ 中等：600 年修正规则逻辑上永不会触发

**问题描述**（`src/engine/solar.ts:13-23`）：

```typescript
const isBasicLeap = [1, 2, 4, 5, 7].includes(remainder)
if (isBasicLeap && year % 600 === 0) return false
```

能被 600 整除的年份，其对 8 取余必为 0（600 = 75×8）。余数 0 不在 `[1,2,4,5,7]` 中，因此 `isBasicLeap` 恒为 `false`，修正分支**永不会执行**。

虽然由于问题 3.1 的存在导致实际效果存疑，但这段代码本身是一个"死代码"逻辑问题。即使将来修正了平年天数，这条规则仍然不会生效。

**建议**：如果确实设计为"能被 600 整除的年份从闰年改为平年"（而非"原本就是平年"），则需要修改基本闰年列表使能被 600 整除的年份原本是闰年，或调整修正逻辑。

### 3.4 ★★ 中等：潮汐等级计算逻辑复杂且有隐藏缺陷

**问题描述**（`src/engine/astronomical.ts:160-182`）：

当前实现使用了一个三层嵌套条件判定中潮，代码可读性差：

```typescript
if (distanceFromAlignment <= 3) {
  return TideLevelEnum.SPRING
} else if (distanceFromAlignment * 2 <= halfCycle && normalized <= halfCycle / 2 + 6) {
  const quarterDistance = Math.abs(normalized - halfCycle / 2)
  if (quarterDistance <= 4.5) {
    return TideLevelEnum.NEAP
  }
}
return TideLevelEnum.NORMAL
```

**问题**：
1. 外层条件的意义不清晰：`distanceFromAlignment * 2 <= halfCycle` 和 `normalized <= halfCycle / 2 + 6` 的逻辑难以从物理上解释
2. 中潮判定使用硬编码的 ±4.5 天窗口，与双月会合周期的比例关系不明确
3. **可能漏判情况**：当 `normalized` 接近 `halfCycle/2 = 9.276`（弦位置）时，`distanceFromAlignment` = min(9.276, 9.276) = 9.276，此时 `9.276 * 2 = 18.552 <= 18.552` ✓，`9.276 <= 15.276` ✓，进入中潮判定。但当 `normalized` 在 [3, 4.776] 范围时，`distanceFromAlignment` 较小（≤3 判定为大潮），但在此区间的边缘（约 4-5 天），实际潮汐可能更接近中潮而非大潮

**建议**：重构为更清晰的基于角度的判定逻辑：
```
angle = (phaseInCycle / SYNODIC_CYCLE) * 360  // 双月黄经差（度）
if angle near 0° or 180° (± tolerance)  → 大潮
if angle near 90° or 270° (± tolerance) → 中潮
else                                     → 小潮
```

### 3.5 ★★ 中等：副月历未实现闰月系统

**问题描述**（`src/engine/lunar-secondary.ts:92-98`）：

```typescript
return {
  year, month, day,
  isLeapMonth: false,  // 恒为 false
  phase,
}
```

设计文档（`历法.md`）明确规定副月历应同样采用"无中气置闰法"：1 回归年 ≈ 8.45 个副月，每 2 年需自动生成 1 个闰月。

当前实现中 `isLeapMonth` 始终为 `false`，导致副月历无法正确与回归年对齐——长期运行下副月月份会逐步漂移。

**建议**：参照主月历的 `hasZhongQi` 逻辑实现副月历的中气检测与闰月判定。

### 3.6 ★ 轻微：节气检测采用简单均匀分布

**问题描述**（`src/engine/astronomical.ts:24-43`）：

24 个节气被假设为在回归年内均匀分布（间距 = 350.6266/24 ≈ 14.60944 天）。

**实际问题**：真实天文学中，由于地球轨道为椭圆形，太阳在黄道上的视运动并非匀速。节气间的实际间隔在 14.7-15.2 天之间波动（以地球为例）。

**当前影响**：在虚构的双月行星系统中，这也取决于轨道偏心率。如果地星轨道近圆形（这是世界设定未说明的），则均匀分布是合理的近似。但若是椭圆轨道，节气检测会有最长约 0.5 天的偏差。

**建议**：在项目文档中明确地星轨道的偏心率假设；若偏心率不可忽略，改用基于黄道经度的精确节气计算。

### 3.7 ★ 轻微：干支纪日基准点的特殊性

**问题描述**（`src/engine/ganZhi.ts:17`）：

```typescript
const adjusted = abs - 1
```

`ABS=0` 映射到干支循环的前一天位置（癸亥·第 60 天），`ABS=1` 才是甲子·第 1 天。这意味着公元 0 年第 1 天（ABS=0）被定义为"癸亥日"而非"甲子日"。

虽然测试验证了这一行为，但设计文档未明确说明为何选择这一偏移。可能存在两种解释：
- 公元元年前一天恰好是甲子日（ABS=1 是甲子）
- 有意使干支循环不与公元纪元对齐

**建议**：在代码注释或文档中明确此偏移的设计理由。

### 3.8 ★ 轻微：累积舍入法的精度边界

**问题描述**（`src/engine/lunar-primary.ts:51-53`）：

```typescript
function cumulativeDaysBeforeMonth(globalMonthIndex: number): number {
  return Math.round(globalMonthIndex * SYNODIC_MONTH)
}
```

`Math.round()` 使用 IEEE 754 双精度浮点运算。对于大索引值（如 globalMonthIndex > 10^7，对应约 50 万年），浮点乘法会累积精度误差，导致 `Math.round()` 结果偏移。

**实际影响**：在项目支持的 0-5000 年范围内（约 9 万个主月），精度完全无问题。此问题仅在超长期运行（>10 万年）时需关注。

**建议**：当前无需修改，但可在代码注释中标注适用年限范围。

## 四、结论

### 4.1 整体评价

双月合历项目是一个**设计精良、实现严谨的历法引擎系统**，具有以下突出优点：

1. **架构清晰**：engine 层作为纯函数核心，与 UI 层完全解耦。所有历法转换以 AbsoluteDayNumber 为唯一坐标基准，实现了三轨历法的统一表达
2. **测试充分**：6 个测试文件涵盖单元测试、集成测试、双向转换一致性验证，测试用例超过 200 个，覆盖了关键边界
3. **科学严谨**：历法规则直接基于天体力学周期参数推导，阳历、主月历、副月历各有清晰的设计依据
4. **工程化成熟**：TypeScript 严格类型、Zustand 状态管理、预计算缓存机制（同步/异步双模式）、暗色主题适配等
5. **文化设计完整**：独立的 5 日制星期命名（人天/兽天/碧森天/岩矿天/龙天）、60 干支纪日、二十四节气命名，形成了完整的文化体系

### 4.2 关键缺陷

存在一个**严重影响核心功能正确性**的问题（问题 3.1）：**阳历平年天数设为 349 天**，导致年平均长度仅为 349.625 天，与回归年 350.6266 天相差约 1 天/年。这意味着历法的季节定位功能会每年漂移 1 天，8 年累积漂移 8 天，完全丧失了"阳历定四季"的基础能力。修正方案明确且工作量小（修改 2 个常量 + 调整 12 月平年天数）。

另有两个**中等影响**的缺陷：月相区间的整数空隙（问题 3.2）会导致部分日期的月相图标显示错误；副月历未实现闰月（问题 3.5）会导致长期运行下副月月份漂移。

### 4.3 综合评价

该项目作为一套虚构世界的完整历法系统，**核心算法设计合理，工程实现质量高**。三轨协同机制通过 ABS 坐标基准和中气绑定实现了有机耦合而非简单堆砌。当前存在的问题主要集中在常量配置层面（问题 3.1、3.2），而非架构层面，修复成本低、收益高。修复上述关键问题后，该系统可作为完整的生产级历法引擎使用。
